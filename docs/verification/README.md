## Funnel redesign (planning/50) — how it was verified

The story-creation flow was rebuilt (see `planning/50-funnel-redesign.md`): the
child's name is split into 성/이름 (the story uses the given name only), tonight's
subject accepts a **free-text seed alongside the situation chips**, the awkward
place picker is **removed**, characters come from a **reusable roster (보관함)**,
and the flow is a **3-step funnel with a progress bar** (topic → cast → tone)
whose state survives native back-swipe.

### Against the REAL backend + real AI (kagi)

The same on-device journey was run pointing at the real Playzy backend (which
proxies `kagi serve` → the "Playzy 동화작가" custom-assistant profile), so the
story is generated by a real LLM, not a fake:

```bash
# terminal 1: kagi serve (auto-login via KAGI_EMAIL/PASSWORD)
kagi serve --addr 127.0.0.1:8921
# terminal 2: playzy backend, routed through the story-author profile
KAGI_PROFILE_ID=<uuid> go run ./backend
# terminal 3: drive the app on the sim against that backend
flutter test integration_test/app_journey_test.dart -d "iPhone 17" \
  --dart-define=PLAYZY_API_BASE_URL=http://127.0.0.1:8080
```

**Result: All tests passed** — the full flow (name entry → SDUI catalog → options
→ generation) ran through the real backend and the reader rendered a real,
personalized, night-mode story. [ios/ios-04-real-backend-home.png](ios/ios-04-real-backend-home.png)
shows the app on the simulator with the allowance card populated from the real
backend `/v1/quota` (not a fake).

Notes:
- iOS ATS blocks cleartext HTTP by default, so `Info.plist` carries
  `NSAllowsLocalNetworking` (loopback/local only; real domains still require
  HTTPS — production uses an HTTPS API).
- A pixel screenshot of the in-test reader isn't reliable: `flutter test
  integration_test` doesn't keep the app foregrounded for `simctl` (it captures
  the springboard), and synthetic keyboard input can't reach the field to drive
  a fresh reader by hand. The behavioral guarantee is the passing on-device test
  above; the story content itself was verified directly through the backend
  chain (valid JSON, exact page count, injection/unsafe probe resisted).

## How to reproduce

```bash
# tests + build (in the Docker toolchain)
./app/tool/flutter.sh test
docker run --rm -v "$PWD/app:/app" -w /app ghcr.io/cirruslabs/flutter:stable \
  flutter build web --release

# serve + drive (host)
python3 -m http.server 8099 --directory app/build/web &
#   then a headless browser over http://localhost:8099 (see the flow above)
```

Note: the app renders with CanvasKit (canvas, not DOM), so the capture drives by
pointer coordinates. The behavioral guarantee is the `app_journey_test.dart`
integration test; these screenshots are the visual confirmation.
